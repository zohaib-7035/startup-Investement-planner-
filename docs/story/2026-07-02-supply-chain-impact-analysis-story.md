## Story 13: Traverse a Financial Relationship Graph to Identify Disruption Impact

**As a** financial RAG engineer building the investment intelligence pipeline,
**I want** a function that accepts a list of financial relationship triples and an entity name, traverses the graph using breadth-first search to discover all directly and indirectly affected entities, and returns a structured result with the affected entities grouped by proximity and a step-by-step reasoning chain,
**So that** the platform can answer supply-chain disruption questions ("if TSMC is disrupted, who else is affected and why?") without calling an LLM — using only the structured knowledge graph already extracted by `extract_relationships`.

### Scope In
- Accept a `graph` argument (a list of dicts, each with `source`, `relation`, and `target` string keys — the output format of `data/knowledge_graph.extract_relationships`) and a `disrupted_entity` string as input
- Traverse the graph using breadth-first search starting from `disrupted_entity`, following directed edges in the forward direction (source → target): if entity A is a SUPPLIER to entity B, and A is disrupted, then B is directly affected; any entity that B supplies is then indirectly affected
- Classify each discovered entity as either directly affected (one hop from the disrupted entity) or indirectly affected (two or more hops)
- Build a human-readable reasoning chain as a list of strings — one entry per affected entity — each explaining which path through the graph caused the impact (e.g. "Microsoft is indirectly affected via NVIDIA: TSMC is a SUPPLIER to NVIDIA, and NVIDIA is a SUPPLIER to Microsoft")
- Return a dict with exactly five keys: `disrupted_entity` (string — the input entity name), `directly_affected` (list of strings — entity names one hop away), `indirectly_affected` (list of strings — entity names two or more hops away), `reasoning_chain` (list of strings — one explanation per affected entity in BFS discovery order), and `total_affected` (int — count of all affected entities, directly + indirectly)
- Return a safe fallback dict when: `graph` is empty or not a list, `disrupted_entity` is empty or whitespace, the disrupted entity does not appear as a source in the graph (no entities are affected), or any exception occurs — the fallback must have the same five-key schema with `directly_affected` and `indirectly_affected` as empty lists, `reasoning_chain` as an empty list, and `total_affected` as 0
- Skip any triple in the graph that is missing the `source`, `relation`, or `target` key, or where any of these values is not a non-empty string — do not raise on malformed triples
- Live in `data/graph_reasoning.py` as `analyze_impact(graph: list, disrupted_entity: str) -> dict`
- Covered by `tests/test_graph_reasoning.py` — pure Python, no mocking required (no external calls)

### Scope Out
- No LLM call — the reasoning chain is constructed programmatically from the graph structure, not generated by a language model
- No cycle detection beyond what BFS naturally handles — BFS visits each node once so cycles in the graph do not cause infinite loops
- No relation-type filtering — all relation types (SUPPLIER, CUSTOMER, SUBSIDIARY, PARTNER, etc.) are treated as directed edges when traversing impact
- No reverse traversal — only forward edges (source → target) are followed; if NVIDIA → TSMC exists, disrupting TSMC does not affect NVIDIA through that edge
- No confidence-score weighting — triples are either included or excluded based on schema validity, not filtered by confidence threshold
- No multi-entity disruption scenario — exactly one `disrupted_entity` per call
- No persistence or graph database — the function operates purely on the in-memory list passed as `graph`
- No deduplication of edges in the input graph — if the same triple appears twice, BFS handles it gracefully because each node is visited at most once

### Acceptance Criteria
- Given the graph `[{source:TSMC, relation:SUPPLIER, target:NVIDIA}, {source:TSMC, relation:SUPPLIER, target:AMD}, {source:NVIDIA, relation:SUPPLIER, target:Microsoft}]` and disrupted entity "TSMC", when `analyze_impact` is called, then `directly_affected` contains "NVIDIA" and "AMD", `indirectly_affected` contains "Microsoft", and `total_affected` is 3
- Given the same graph and disrupted entity "TSMC", when `reasoning_chain` is inspected, then it contains exactly 3 entries — one for each of NVIDIA, AMD, and Microsoft — and the Microsoft entry references the path through NVIDIA
- Given a successful call, when the returned dict is inspected, then it contains exactly five keys: `disrupted_entity`, `directly_affected`, `indirectly_affected`, `reasoning_chain`, `total_affected`
- Given a disrupted entity that does not appear as a source in any triple (e.g. "Apple"), when `analyze_impact` is called, then `directly_affected` and `indirectly_affected` are both empty lists, `reasoning_chain` is an empty list, and `total_affected` is 0 — and no exception is raised
- Given an empty graph list `[]`, when `analyze_impact` is called, then the function returns the safe fallback dict without raising
- Given a graph that contains a malformed triple missing the `source` key, when `analyze_impact` is called, then the malformed triple is skipped and valid triples are still processed correctly
- Given a graph with a cycle (e.g. A → B → C → A), when `analyze_impact` is called with disrupted entity "A", then B and C are returned as affected and the function terminates without looping infinitely
- Given `total_affected` is inspected, when compared against the sum of `len(directly_affected)` and `len(indirectly_affected)`, then the values are equal

### Definition of Done

- [ ] BFS traversal correctly classifies entities at depth 1 as directly affected and entities at depth 2+ as indirectly affected
- [ ] Reasoning chain entries include the path description for each affected entity
- [ ] Malformed triples (missing keys or non-string values) are skipped without raising
- [ ] Outer `try/except Exception` wrapping the entire function body returns the safe fallback dict on any uncaught error
